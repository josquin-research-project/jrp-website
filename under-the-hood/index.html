Josquin Research Project: Infrastructure Overview

The Josquin Research Project consists of two primary components:

  • The website
  • The data server and digital score files

Together these two components provide public access front end to encoded musical scores, metadata, and derived analytical resources.

Website

The public website for the project is hosted using GitHub Pages, which provides static web hosting built on Jekyll and Liquid templates.

Repository:

https://github.com/josquin-research-project/jrp-website

GitHub Pages automatically builds the site whenever changes are committed to the repository. Updating the website therefore requires only committing modifications to the repository; the site rebuilds and deploys automatically.

Advantages include:

  • Collaborative editing through pull requests
  • Full version control through Git history
  • Automatic publishing after commits
  • Global distribution through GitHub's content network
  • No server maintenance requirements
  • Free hosting

Feature requests and bug reports can be submitted through the Issues system:

https://github.com/josquin-research-project/jrp-website/issues

All changes to the website are tracked in the commit history:

https://github.com/josquin-research-project/jrp-website/commits/main

Data Server

The Josquin Research Project also operates a separate data server that manages digital scores, format translations, and prepared analytical data:

https://data.josqu.in

The server provides API endpoints that allow retrieval of musical data in multiple formats, including:

  • Humdrum
  • MEI
  • MusicXML
  • MIDI
  • analytical graphics

The primary archival format used for encoding musical scores is the Humdrum format.

All encoded scores are maintained in the repository:

https://github.com/josquin-research-project/jrp-scores

These files contain the authoritative digital transcriptions encoded and used by the project.

The data server automatically imports the contents of this repository into its internal framework. The server-side infrastructure that performs this processing is maintained in the repository:

https://github.com/craigsapp/jrp-data-server

This software handles:

  • data indexing
  • format conversion
  • API responses
  • generation of derived files such as incipits and analytical datasets

Further documentation is available in the repository README file.

Example Data Object

Example incipit file:

https://data.josqu.in/Agr1001a-incipit.svg

Filename interpretation:

  • Agr --- composer Alexander Agricola
  • 1001 --- work number
  • a --- first section of the work

The section letter typically corresponds to movements of multi-section works (for example Mass movements such as Kyrie, Gloria, Credo, Sanctus, Agnus).

List of composer codes:

https://bit.ly/renaissance-composer-codes

Encoding Standards

Encoding standards are described here:

https://wiki.ccarh.org/wiki/Josquin

These standards define how Renaissance musical notation is represented within the Humdrum framework. Topics include:

  • part naming conventions
  • mensural notation representation
  • editorial accidentals
  • cadence markings
  • section boundaries
  • text underlay conventions
  • metadata documentation

These conventions ensure consistency across scores encoded by multiple contributors.

File Naming System

Each musical work uses a structured identifier:

CCC####S

Where:

  • CCC — three-letter composer code
  • #### — four-digit work number
  • S — optional section letter

Examples:

Jos2303 Agr1001a Obr2005b

Explanation:

  • The composer code identifies the composer.
  • The four-digit number identifies the work.
  • The optional letter identifies a section or movement.

Master Metadata Index

All work metadata is maintained in a master spreadsheet:

https://bit.ly/renaissance-metadata

This spreadsheet contains catalog information such as:

  • composer
  • work title
  • manuscript sources
  • catalog numbers
  • scoring
  • editorial metadata

A Google Apps Script converts the spreadsheet into JSON data used by the website:

https://script.google.com/macros/s/AKfycbwuWnQ6O9AGgbBT7B2rLPwEQw9i7NRHtkA2Jq8A1L6drL9Th7PR-jARwDKP6kZEhED56w/exec?sheet=works

The website loads this JSON file dynamically to support:

  • metadata searching
  • browsing by composer
  • work listings
  • catalog navigation

This architecture allows the project to maintain a single authoritative metadata source while automatically distributing updated data to the public website.
